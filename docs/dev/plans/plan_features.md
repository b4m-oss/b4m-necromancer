# 追加機能アイデア・優先度メモ

このドキュメントは、b4m-necromancer に今後追加しうる機能アイデアを整理したものです。  
実装順や詳細仕様はまだ未確定ですが、「何をやると嬉しいか」と「おおまかな設計の方向性」をメモします。

---

## 1. 優先度の整理（★ は暫定目安）

- **テストモード（ドライラン）** ★★★★
- **モードごとのアップロード先切り替え** ★★★
- **再送キュー** ★★★
- **簡易ヘルスチェックコマンド** ★★★
- **ログレベル切り替え** ★★★
- **ファイル名ルール** ★★
- **--dump-config** ★★
- **ローテーション（tmp／ログ削除ポリシー）** ★
- **OCR 連携（Webhook 的フック）** 未採点
- **簡易 Web UI** 低め（ただしあると嬉しい）

ここでは、特に優先度の高そうなものを中心に設計メモを残します。

---

## 2. テストモード（ドライラン） ★★★★

### 目的

- 設定変更や新しいアップロード先（Dropbox / Google Drive など）を導入する前に、  
  **「実際にはスキャン・アップロードせず、どのように動くかを確認したい」**。
- 実機スキャンやクラウドへの書き込みを行わないことで、安全に試運転できるモードを提供する。

### 方針（実装済み仕様）

- `scan.py` の CLI に `--dry-run` オプションを追加。
  - `scan.py --dry-run check`
    - スキャナ検出 (`scanimage -L` ベース)・ウォームアップ (`scanimage -n` ベース)・Nextcloud エンドポイントへの HTTP 接続テストを **実スキャン・アップロードなし** で実行。
    - 結果を `Scanner: OK/NG / Warm-up: OK/NG / Cloud: OK/NG` 形式でサマリ表示し、どれか 1 つでも NG なら非 0 終了コードを返す。
  - `scan.py --dry-run scan <mode>`
    - 上記と同じスキャナウォームアップロジックを通した上で、指定モード（例: `diary`）の解像度・モード・給紙設定を使い、**A4 片面カラー 1 枚のみ** をテストスキャン。
    - 出力先は `tmp/DRYRUN-YYYYMMDDHHMMSS/<mode>-dryrun-1.jpg` のような一時ディレクトリで、クラウドへのアップロードや `delete_after_upload` による削除は一切行わない。
- `--dry-run` は次のオプションと排他的:
  - `--dump-config`, `--list`, `--output`, 通常のバッチスキャン（`config` 単独指定）と同時には使えない。
  - 不正な組み合わせ時はエラーメッセージを標準エラーに出力して終了する。

### 実装のポイント

- ローカル疎通チェック:
  - `ScannerManager.check_scanner_available()` により、`scanner.json` の設定に基づいて利用可能なスキャナを検出。
  - `ScannerManager.warm_up_scanner()` により、`scanimage --device=... -n` を実行してウォームアップ状態を確認。
  - Nextcloud 側は `test_nextcloud_connection()` で、`upload.json` の `nextcloud.endpoint` に対して認証付き `curl -I`（HEAD）を投げ、HTTP ステータス 2xx〜3xx を OK とみなす。
- 1 枚テストスキャン:
  - `load_scan_configs()` / `load_scanner_config()` を使って、実運用と同じ解像度・モード・給紙モードを利用。
  - 用紙サイズはコード側で A4 固定（幅 210mm、高さ 297mm）として `--page-width` / `--page-height` を指定。
  - `scanimage` の実行結果（stdout / stderr / return code）をログに出し、出力ファイルの存在とサイズを検証したうえで成功/失敗を判定。
- いずれのドライランモードでも「実際のアップロード処理 (`uploader.upload_*` 系) や tmp ディレクトリ削除」は発生せず、安全に動作確認のみを行える。

## 3. モードごとのアップロード先切り替え ★★★

### 目的

- `diary` / `receipt` / `flyer` など、モードごとに **クラウド上の保存先フォルダを分けたい**。
  - 日記: `Scans/Diary/...`
  - レシート: `Scans/Receipt/...`
  - フライヤー: `Scans/Flyer/...`

### 方針（イメージ）

- `mode.json` に、モードごとの `upload_subfolder` を追加する案:

```json
"diary": {
  "mode": "color",
  "resolution": 200,
  "source": "ADF Duplex",
  "file_format": "pdf",
  "upload_subfolder": "Diary/",
  ...
}
```

- Uploader 側（Nextcloud / 将来の Dropbox / GDrive）で、
  - ベースの `upload_folder`（例: `"Scans/"`）と
  - モードの `upload_subfolder`（例: `"Diary/"`）
  を結合して、最終的な保存先ディレクトリを決定する。

### 実装のポイント

- 「モード名 → サブフォルダ」の解決は、`scan.py` 側で決めてから Uploader に渡すか、  
  あるいは、Uploader に `subfolder` 引数を追加するかは要検討（後者の方がクラウド別の挙動を調整しやすい）。
- Nextcloud 以外のクラウドでも同じ考え方を採用できるよう、  
  パス結合ロジックは Uploader 側に寄せる方が良さそう。

## 4. 再送キュー ★★★

### 目的

- ネットワーク不調やクラウドサービス側の一時的な障害でアップロードが失敗した場合に、  
  **「あとから安全に再送できる仕組み」** を持ちたい。

### 段階的な方針

1. **フェーズ1: 失敗メタ情報の記録のみ**
   - アップロードが失敗したとき、以下の情報を JSON などで `/var/log/scanner/failed_uploads/` などに保存:
     - ローカルファイルパス（またはディレクトリパス）
     - モード名、タイムスタンプ
     - アップロード先情報（クラウド種別、リモートパス）
     - エラー内容（HTTP ステータス、例外メッセージ等）

2. **フェーズ2: 再送スクリプト**
   - `tools/retry_failed_uploads.py` のような CLI ツールを用意し、
     - `failed_uploads/` ディレクトリを読み込み、
     - 1 件ずつ再送を試みる。
   - 成功したものはメタ情報を削除、失敗は残す。

### 実装のポイント

- 本体のスキャンフロー（`scan.py`）には最初は「失敗メタ情報を書き出す」だけ入れておき、  
  再送処理は別プロセス／別ツールとして切り離すと構造がシンプルになる。

## 5. 簡易ヘルスチェックコマンド ★★★

### 目的

- 「何かおかしい」と感じたときに、**一発で状態確認できるコマンド** を用意する。
  - スキャナが OS に見えているか
  - 設定ファイルが有効な JSON か
  - 最低限のクラウド設定ができているか

### 方針（イメージ）

- `scan.py` に `--selfcheck` オプションを追加。
- 実行内容の例:
  1. `scanimage -L` を実行して、`scanner.json.device_name` がリストに含まれているか確認。
  2. `scanner.json` / `mode.json` / `upload.json` の読み込み・バリデーション。
  3. `upload.json.provider` に応じた簡易チェック（Nextcloud なら `endpoint` 末尾 `/` の有無など）。
  4. 結果を「OK / NG」と、NG の場合の理由一覧として表示。

### 実装のポイント

- 実際のスキャン・アップロードは行わないため、安全に実行できる。
- 将来的に `--dump-config` と連携させ、「このモードで実際に組み立つコマンドも表示する」拡張も検討可能。

---

## 6. ログレベル切り替え ★★★

### 目的

- 平常時はログを控えめ（INFO）、トラブルシュート時だけ詳細（DEBUG）にすることで、  
  ログのノイズを抑えつつ、必要なときにだけ深掘りできるようにする。

### 方針（イメージ）

- どこか 1 箇所（例: 将来的な `hardware.json` / `device.json` / 既存の `scanner.json`）に:

```json
{
  "log_level": "INFO"
}
```

のような項目を持たせる。

- `keypad_daemon.py` の `setup_logging` でこの値を読み取り、  
  `logging.basicConfig(..., level=...)` に反映する。

### 実装のポイント

- 現状 `keypad_daemon.py` はすでに `logging` を使っているため、  
  「設定ファイルから log_level を読む」だけなら影響範囲は限定的。
- `scan.py` 側はまだ `print` が中心なため、必要に応じて:
  - 一部を `logging` に寄せる
  - もしくは `--verbose` オプションのような形で段階的に対応する
  といったステップを踏む想定。

## 7. ファイル名ルール ★★

### 目的

- スキャン結果のファイル名を、**日付・モード・連番などを含む分かりやすい形式にしたい**。
  - 例: `diary-20250303-001.pdf`, `receipt-20250303-01.jpg` など。

### 方針（イメージ）

- シンプルなパターン文字列をどこかに持たせる:
  - 例: `"{mode}-{date}-{seq}.pdf"` のような表現。
- 実装コストと柔軟性のバランスを取るため、最初は:
  - 固定パターン（`{mode}-{timestamp}.pdf` 程度）から始めてもよい。
  - あるいは「今のタイムスタンプディレクトリ＋既存ファイル名」で運用し、  
    実際の不便が見えるまで後回しにする選択もあり。

---

## 8. `--dump-config` ★★

### 目的

- 将来モードやクラウド設定が複雑化したときに、  
  **「最終的にどのコマンド・どのパラメータでスキャン・アップロードされるか」** を一発で確認したい。

### 方針（イメージ）

- `scan.py --dump-config <mode>` のような形で:
  - `scanner.json` / `mode.json` / `upload.json` を読み込んだうえで、
  - 実際に `scanimage` に渡すコマンドライン、
  - Uploader に渡すアップロードパス情報
  を整形して出力する。

### 実装のポイント

- 上述の `--selfcheck` の一部として同等の情報を出力する設計にすると、  
  コマンドが増えすぎずに済む。

---

## 9. ローテーション（tmp / ログ削除） ★

### 目的

- 長期運用時に、一時ディレクトリやログが溜まりすぎてストレージを圧迫するのを防ぐ。

### 方針（イメージ）

- `upload.json` などに:

```json
{
  "local_retention_days": 14
}
```

のような設定を持たせ、  
`tools/cleanup_old_files.py` のようなスクリプトで「N 日以上前の tmp ／ログを削除」する。

### 備考

- 実際にどの程度ファイルが溜まるかをしばらく観察してから着手しても問題なさそうなため、優先度は低め。

---

## 10. OCR 連携（Webhook 的フック）

### 目的

- 本体で OCR 機能を持つのではなく、**外部 OCR や別ツールに処理を委ねるフック** を提供する。
  - 例: スキャン後に自前の OCR スクリプトやクラウド OCR サービスを叩く。

### 方針（イメージ）

- `upload.json`（もしくは別設定）に、任意のフック設定を追加:
  - 例1: `post_scan_command`: `"bash ocr_and_rename.sh {pdf_path}"`
  - 例2: `webhook_url`: `"https://example.com/webhook"` を HTTP POST する。
- スキャン完了後（アップロード前 or 後）に、このフックを呼び出すポイントを `scan.py` に用意。

### 備考

- どのタイミングで呼ぶか（アップロード前 / 後）、  
  失敗時に本体のフローを止めるかどうかなど、細かい仕様は未確定。

## 11. 簡易 Web UI（低優先）

### 目的

- ブラウザから:
  - 現在の状態（daemon 稼働中か、最後のスキャン時刻など）
  - 最近のログ／スキャン履歴
 などを軽く確認できると便利。

### 方針（イメージ）

- 非常に小さな Web サーバ（Flask / FastAPI など）を追加し、
  - `/status` で状態・設定のサマリ
  - `/logs` で直近ログの tail
  程度を表示する。
- Raspberry Pi Zero 2 W のリソースを考慮し、
  - 更新頻度の低いポーリング前提
  - 認証などは LAN 内利用前提で簡素に
  といった制約をかける。

### 備考

- 優先度は低く、「運用が安定してきて、遊びとして実装したくなったら」という位置づけ。

---

## 12. テスト戦略（自動テスト）

### 目的

- 設定や機能を拡張していっても、**既存の挙動が壊れていないことを自動で確認したい**。

### 方針（イメージ）

- 単体テスト（pytest など）を中心に、次のような領域をカバーする:
  - `scan.py`
    - `scanimage` コマンドラインの組み立てロジック（モード別オプション、page-height, ald など）。
    - `is_blank_page` の閾値まわり（簡単なテスト画像またはモック）。
  - `nextcloud.py` / 将来の Uploader
    - URL / パスの組み立てロジック。
    - HTTP レスポンスの解釈（ステータスコード解析）。
  - 設定ファイル読み込み関数
    - `mode.json` / `scanner.json` / `upload.json` のマージ・デフォルト補完ロジック。

- 実際のハード依存部分（scanimage 実行、GPIO 操作）は:
  - コマンド文字列の検証に留める、
  - あるいはモック／スタブを使う、
  といった形でテスト可能な部分と切り離す。

---

## 13. まとめ

- まずは **テストモード / モード別アップロード先 / ログレベル / ヘルスチェック / 再送メモ** あたりを優先的に検討すると、  
  「運用しながら安心して拡張できる」度合いが大きく向上する。
- その上で、LED・ブザー・追加クラウド対応・OCR フックなどを順次実装していくと、  
  小さく始めたプロダクトが、かなり本格的な「スキャナ延命プラットフォーム」になっていくはず。
