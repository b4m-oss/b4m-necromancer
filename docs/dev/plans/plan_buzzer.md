# ブザー（パッシブブザー）追加 設計メモ

本ドキュメントは、b4m-necromancer に「ブザーによるサウンド通知」を追加する際の、ざっくりとした方針メモです。  
現時点では実装は行わず、アイデアレベル・設計レベルの整理だけを行います。

---

## 1. 目的・ユースケース

- スキャン完了やエラー発生を **音** で知らせる。
  - スキャン完了（＋クラウドアップロード成功）時に「成功音」。
  - 何らかのエラー（スキャン失敗・アップロード失敗など）が発生したときに「エラー音」。
- 小型スピーカーではなく、**パッシブブザー** を使って簡易的なメロディ／トーンを鳴らす。
  - 例:
    - 成功: 高めの音が上昇する 2〜3 音のパターン（ピッ・ピピッ）
    - 失敗: 低めの音が下降するパターン（ブー・ブッ）

---

## 2. ハードウェア方針（ふわっと案）

- Raspberry Pi の GPIO にパッシブブザーを接続。
  - パッシブブザーは PWM 信号により音程を制御可能。
  - GPIO 番号は設定ファイルで変更可能にする（デフォルト値はひとつ決める）。
- 電気的には、ブザー単体 or トランジスタ＋ダイオード構成などを前提（消費電流に応じて要検討）。  
  → 具体的な回路（保護素子の要否など）は実装時に確認（**未確定**）。

## 3. ソフトウェア設計方針

### 3.1 ブザー制御モジュール

- 例: `app/lib/buzzer.py` を新設。
- 役割:
  - GPIO / PWM の初期化と制御を一元化。
  - 「成功音」「エラー音」などを、関数単位で提供する。

```python
class Buzzer:
    def __init__(self, gpio_pin: int):
        ...

    def play_success(self):
        # 短く上昇するトーンパターン
        ...

    def play_error(self):
        # 下降系や不協和なトーンなど、エラーと分かるパターン
        ...

    def shutdown(self):
        ...
```

- 実装上は、周波数と長さのシーケンスを定義しておき、  
  例: `[(freq1, dur1), (freq2, dur2), ...]` を順に鳴らすイメージ。

### 3.2 連携ポイント

- 主に `keypad_daemon.py` / `scan.py` との連携を想定。
  - スキャン開始時:
    - 音は不要 or 短い「ピッ」程度（任意）。
  - スキャン＋アップロード正常終了時:
    - `Buzzer.play_success()` を呼ぶ。
  - スキャンエラー / アップロードエラー時:
    - `Buzzer.play_error()` を呼ぶ。

> メモ:  
> 実装時には、「スキャンだけ成功したがアップロードが失敗した」「白紙判定で全ページが消えた」など、  
> 細かい状態の違いをどこまで音パターンに反映するかは別途検討（まずは成功 / 失敗の 2 種類で十分）。

## 4. 設定ファイルとの連携（イメージ）

- LED と同様に、将来的にハードウェア設定用のファイル（例: `hardware.json`）にまとめる案:

```json
{
  "led": {
    "enabled": true,
    "gpio_pin": 18,
    "active_high": true
  },
  "buzzer": {
    "enabled": true,
    "gpio_pin": 19,
    "active_high": true
  }
}
```

- 当面はハードコードでもよいが、最終的には:
  - ブザーを無効化したいユーザー向けに `enabled` フラグを提供。
  - GPIO 番号を変更できるようにして、環境差を吸収。
