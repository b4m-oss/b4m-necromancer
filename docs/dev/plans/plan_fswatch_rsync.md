## b4m-necromancer ローカル開発環境用 rsync 自動同期仕様書（fswatch 利用）

### 目的

- **Mac 上の作業ディレクトリ `b4m-scan` の変更を自動検知し、RPZ2W（`sumoto.local`）上の `b4m-necromancer` ディレクトリへ rsync で自動転送する仕組み**を整える。
- 手動 rsync による同期忘れを減らし、ローカル開発と RPZ2W 実機動作確認のサイクルを短くする。

---

## 全体像

- **監視ツール**: macOS 上で `fswatch` を使用し、`b4m-scan` ディレクトリ配下の変更を検知する。
- **転送ツール**: 検知のたびに `rsync` を実行し、RPZ2W 上の `/home/kohki/b4m-necromancer` と同期する。
- **実行形態**:
  - 基本は **バックグラウンドジョブ**として動かし、ターミナルを占有しない。
  - 必要に応じて **フォアグラウンド実行（デバッグ用途）** も可能とする。
- **操作インターフェイス**:
  - シェルスクリプトで「起動」「停止」を提供する。
  - 最終的には `Makefile` のターゲットから呼び出せるようにする。

---

## 前提条件

- **ローカル（Mac）側**
  - Homebrew で `fswatch` がインストールされていること。
    - 例: `brew install fswatch`
  - プロジェクトルート:
    - `/Users/kohki/projects/ビジネス/b4m-scan`
  - このリポジトリに `Makefile` および本仕様書の Markdown を配置する。

- **リモート（RPZ2W = sumoto.local）側**
  - ホスト名: `sumoto.local`
  - ユーザ: `kohki`
  - ベースディレクトリ: `/home/kohki`
  - 同期先ディレクトリ: `/home/kohki/b4m-necromancer`
    - `rsync` が自動的に作成することを想定するため、事前作成は必須ではない。
    - ただし `/home/kohki` 自体は存在している前提。

- **SSH 接続**
  - `ssh sumoto.local` でパスワード入力のみ、もしくは公開鍵認証で接続可能な状態になっていること。
  - `rsync` コマンドが `ssh` 経由で使用できること（標準的な環境を前提）。

## rsync 設計

### 同期元・同期先

- **同期元（ローカル）**
  - `/Users/kohki/projects/ビジネス/b4m-scan/`
  - 末尾に `/` を付け、**ディレクトリ自体ではなく「中身」を同期**対象とする。

- **同期先（リモート）**
  - `sumoto.local:/home/kohki/b4m-necromancer/`
  - 末尾に `/` を付け、ここに同期元の中身を流し込む。

### 基本 rsync オプション

- `-a`: アーカイブモード（パーミッション等をできる限り保持）
- `-v`: 詳細ログ出力
- `-z`: 転送時に圧縮
- `--delete`: 同期元から削除されたファイルを同期先からも削除し、「鏡像」を維持

### 基本コマンド例

```bash
rsync -avz --delete \
  "/Users/kohki/projects/ビジネス/b4m-scan/" \
  "sumoto.local:/home/kohki/b4m-necromancer/"
```

- 実行したタイミングでのみ差分を計算・転送する。
- 自動で実行したい場合は、後述の `fswatch` 連携または `Makefile` ターゲット経由で起動する。

## fswatch + rsync 連携仕様

### 監視対象

- ディレクトリ:
  - `/Users/kohki/projects/ビジネス/b4m-scan`
- 監視範囲:
  - 再帰的監視（サブディレクトリも含めて監視）

### 監視コマンドの基本形

```bash
fswatch -r -o "/Users/kohki/projects/ビジネス/b4m-scan" | while read; do
  rsync -avz --delete \
    "/Users/kohki/projects/ビジネス/b4m-scan/" \
    "sumoto.local:/home/kohki/b4m-necromancer/"
done
```

- `fswatch -r -o DIR`
  - `-r`: 再帰的監視
  - `-o`: 何らかの変更があったら「1 行だけ」イベントを出力（大量イベントをまとめる用途）
- `while read; do ... done`
  - イベント 1 回ごとに `rsync` を実行するループ。

### バックグラウンド実行（仕様）

- 通常運用では、上記監視ループを **バックグラウンドジョブ**として起動する。
- 代表的な実行形態:

  - シェルスクリプトから:

    ```bash
    <監視 + rsync を行うコマンド> &
    ```

  - もしくはターミナルを閉じても動かす場合:

    ```bash
    nohup <監視 + rsync を行うコマンド> >/tmp/b4m-fswatch.log 2>&1 &
    ```

- 実際のコマンド列は、後述の「シェルスクリプト仕様」で定義する。

### 停止方法（仕様）

- バックグラウンドで動いている `fswatch`＋`rsync` ループを停止するために、以下の方式のいずれか（または両方）を採用する。

1. **プロセス名ベースでの kill**
   - `ps` コマンド等で `fswatch` プロセス（または専用のラッパスクリプト名）を特定し、`kill` する。

2. **PID ファイル方式**
   - 起動時にプロセス ID（PID）をファイルに保存し、停止時はその PID を読んで `kill` する。
   - この仕様書では、**PID ファイル方式を採用する前提**とする（操作性・自動化のため）。

## シェルスクリプト仕様

### 目的

- `fswatch`＋`rsync` を用いた自動同期を、以下の 2 つのコマンドで操作できるようにする。
  - **起動**: 自動同期をバックグラウンドで開始する。
  - **停止**: 既に起動しているバックグラウンドジョブを安全に停止する。

### 配置想定

- プロジェクトルート直下（`b4m-scan/`）に配置するスクリプトファイルを想定する。
  - 例: `scripts/fswatch_rsync.sh`
  - 実際のファイル名・パスはプロジェクト内で統一して利用する。

### スクリプトのインターフェイス

- コマンドライン引数でモードを指定する。

- 想定インターフェイス:

  - 起動:

    ```bash
    scripts/fswatch_rsync.sh start
    ```

  - 停止:

    ```bash
    scripts/fswatch_rsync.sh stop
    ```

### PID ファイル仕様

- PID 管理のためのファイルを 1 つ用意する。
  - 例: `tmp/fswatch_rsync.pid`
- `start` 時:
  - 既に PID ファイルが存在し、かつその PID が生きている場合は「すでに起動中」と判断し、二重起動を防ぐ。
  - 起動に成功したら、バックグラウンドプロセスの PID をこのファイルに書き込む。
- `stop` 時:
  - PID ファイルを読み、その PID に対して `kill` を行う。
  - 停止後、PID ファイルを削除する。
  - PID が存在しない／すでに死んでいる場合は、その旨を表示し、PID ファイルを削除する。

### start 動作（詳細仕様）

1. 必要なディレクトリ（`tmp/` 等）がなければ作成する。
2. 既存 PID ファイルの有無と生存チェックを行う。
3. 問題なければ、以下のコマンド列をバックグラウンドで起動する。

   ```bash
   fswatch -r -o "/Users/kohki/projects/ビジネス/b4m-scan" | while read; do
     rsync -avz --delete \
       "/Users/kohki/projects/ビジネス/b4m-scan/" \
       "sumoto.local:/home/kohki/b4m-necromancer/"
   done
   ```

   - 実際には `nohup` やログ出力リダイレクトを組み合わせる。

4. バックグラウンドプロセスの PID を PID ファイルに保存する。

### stop 動作（詳細仕様）

1. PID ファイルの存在を確認する。
2. PID を読み取り、そのプロセスが生きているか確認する。
3. 生きていれば `kill` で終了させる。
4. 終了後、PID ファイルを削除する。

## Makefile 統合仕様

### 目的

- 開発者が **`make` コマンドだけで操作できるようにする**ことで、運用の簡便さと手順の統一を図る。

### 想定ターゲット

- **`make sync-on`**
  - `fswatch`＋`rsync` 自動同期を起動する。
  - 内部的には `scripts/fswatch_rsync.sh start` を呼ぶ。

- **`make sync-off`**
  - 起動中の `fswatch`＋`rsync` 自動同期を停止する。
  - 内部的には `scripts/fswatch_rsync.sh stop` を呼ぶ。

- （オプション）**`make sync-once`**
  - `fswatch` を使わず、単発で `rsync` を実行する。
  - 内部的にはスクリプトの `once` モードや直接 `rsync` コマンドを呼ぶ。

### Makefile 内での責務分離

- Makefile はあくまで「入口」とし、実際の処理ロジックはシェルスクリプト側に集約する。
  - メリット:
    - ロジックをシェルスクリプトでテスト／変更しやすい。
    - Makefile 内の記述を最小限に保てる。

---

## 運用フロー（想定）

1. Mac で開発を開始するとき:
   - プロジェクトルートで `make sync-on` を実行。
   - 以後、`b4m-scan` 配下のファイルを保存するたびに、自動で RPZ2W 側へ同期される。

2. 開発終了／同期不要となったとき:
   - プロジェクトルートで `make sync-off` を実行。
   - バックグラウンドで動いていた `fswatch`＋`rsync` プロセスが停止する。

3. 単発で同期したいとき:
   - `make sync-once`（または相当のコマンド）を実行し、その時点の内容だけを同期する。

## 注意事項・今後の拡張余地

- **負荷について**
  - `fswatch` は変更のたびにイベントを出すため、頻繁な保存・大量変更時には `rsync` 実行が多くなる可能性がある。
  - 必要に応じて、`fswatch` のオプションやスクリプト側で「一定時間内の変更をまとめる」などのバッチングを検討する。

- **フィルタリング**
  - `.git/` や一時ファイル、テスト成果物など「同期不要なファイル／ディレクトリ」がある場合、`rsync` の `--exclude` オプション等で制御する。
  - 本仕様書では詳細を定めず、別途 `rsync` 設定の拡張として扱う。

- **永続的な自動起動**
  - 将来的に、macOS の `launchd` を用いてログイン時自動起動とすることも可能。
  - その場合も、基本的なロジックは本仕様書のシェルスクリプトを流用し、`launchd` からスクリプトを呼ぶ設計とする。
