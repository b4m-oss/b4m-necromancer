# pytest 導入準備メモ（関数別テスト設計）

このドキュメントは、b4m-necromancer に pytest ベースの単体テストを導入するためのメモです。  
**「1 関数 1 セクション」** の形式で、

- 各関数／メソッド／ヘルパーが「何をしているか（ステップ）」  
- それに対して考えられる「正常系テスト」「異常系テスト」

を整理しています。  
ここでは **`app/` ディレクトリ配下の Python コード** のうち、テスト優先度が高そうなものを中心にまとめています。

---

## `is_blank_page`

- ステップ
  - 画像ファイルを開き、グレースケールに変換する。
  - 必要に応じてサムネイルサイズ（最大 512×512）に縮小する。
  - ヒストグラムから総ピクセル数と「`white_threshold` 以上」のピクセル数をカウントし、比率を算出する。
  - 白比率が `white_ratio_threshold` 以上なら True（白紙）を返し、例外時は False を返す。

### 正常系テスト

- ほぼ真っ白な画像を渡すと `True` になる。
- 文字や線が多く含まれる画像を渡すと `False` になる。
- 少量のノイズが乗った白背景画像でも、白比率が閾値以上なら `True` になる。

※ 実際のスキャン画像と閾値チューニングに強く依存するため、**単体テスト実装の優先度は低め（ロジック安定後に着手）** とする。

### 異常系テスト

- 存在しないファイルパスを渡した場合に、例外を表に出さず `False` を返す。
- 壊れた画像ファイルを渡しても、例外を表に出さず `False` を返す。

## `create_timestamp_dir`

- ステップ
  - 現在時刻から `YYYYMMDDhhmmss` 形式のタイムスタンプ文字列を生成する。
  - `.../tmp/<timestamp>` ディレクトリを作成（存在しなければ）する。
  - そのディレクトリパスとタイムスタンプをタプルで返す。

### 正常系テスト

- 呼び出すと、存在するディレクトリパスとタイムスタンプ文字列を返す。
- 複数回呼んだ場合に、タイムスタンプ／ディレクトリが基本的に異なる（時間差があれば変わる）。

### 異常系テスト

- 権限のない場所に `tmp` を作ろうとした場合に `OSError` が送出される（環境依存のため、必要であれば統合テスト側で確認）。

## `convert_images_to_pdf`

- ステップ
  - 対象ディレクトリ直下の JPEG / PNG 画像を列挙し、ファイル名順にソートする。
  - 画像が 0 枚ならログを出して False を返す。
  - 画像が 1 枚なら単一ページ PDF、複数枚なら multi-page PDF を生成する。
  - 生成された PDF の存在とサイズ（0 でないこと）を確認し、結果を返す。

### 正常系テスト

- JPEG 1 枚のみの場合に、単一ページ PDF が生成されサイズ > 0 である。
- JPEG 複数枚の場合に、ページ数が期待どおりでエラーなく PDF が生成される。
- PNG を含むディレクトリでも PDF が問題なく生成される。

### 異常系テスト

- 画像ファイルが 0 枚のディレクトリでは False が返り、PDF が生成されない。
- 途中で 1 枚の画像読み込みエラーが発生した場合、False を返し、例外が外に漏れない。

---

## `load_scan_configs`

- ステップ
  - `mode.json` を開き、JSON としてロードする。
  - 読み込んだ dict をそのまま返す。

### 正常系テスト

- 正しい JSON 構造の `mode.json` を読み込める。
- 期待するキー（モード名や `keybindings` など）が含まれていることを確認できる。

### 異常系テスト

- ファイルが存在しない場合に例外が送出される（現状仕様の確認）。
- JSON が壊れている場合に `json.JSONDecodeError` が送出される。

---

## `load_scanner_config`

- ステップ
  - `scanner.json` を開き、JSON として読み込む（存在しない場合や壊れている場合は例外をキャッチ）。
  - iX500 用のデフォルト設定 dict を用意する。
  - 読み込んだ設定でデフォルトを上書きし、不足分はデフォルトで補完する。
  - ファイルが無い／JSON 不正な場合はデフォルト設定をそのまま返す。

### 正常系テスト

- 全フィールドを持つ `scanner.json` を与えたとき、その内容が優先された設定が返る。
- 一部フィールドのみを持つ `scanner.json` を与えたとき、不足フィールドがデフォルトで補完される。

### 異常系テスト

- `scanner.json` が存在しない場合に、デフォルト設定が返される。
- `scanner.json` が壊れた JSON の場合に、エラーメッセージを出しつつデフォルト設定が返される。

## `get_scanner_list`

- ステップ
  - `scanimage -L` をサブプロセスで実行する。
  - 標準出力を文字列として取得して返す。
  - エラー時はエラーメッセージを出力し、空文字列を返す。

### 正常系テスト

- `scanimage -L` が成功し、モックした stdout の内容がそのまま返る。

### 異常系テスト

- `scanimage -L` がコマンドエラーを返すケースで、エラーメッセージを出しつつ空文字列を返す（サブプロセスはモック）。

---

## `scanimage_scan`

- ステップ
  - 引数（出力パス／デバイス名／解像度／モード／ソース／フォーマット）から `scanimage` コマンド配列を組み立てる。
  - サブプロセスとして `scanimage` を実行し、stdout / stderr を取得する。
  - 戻り値が 0 なら成功として True、非 0 ならエラーメッセージを出して False を返す。
  - 例外発生時も False を返す。

### 正常系テスト

- 想定パラメータを渡したとき、組み立てられたコマンド配列が期待どおりである（サブプロセスはモック）。
- サブプロセスが成功（`returncode == 0`）の場合に True を返す。

### 異常系テスト

- サブプロセスが非 0 の戻り値を返した場合に False を返す。
- サブプロセス実行中に例外が発生した場合にも False を返し、例外が外に漏れない。

---

## `monitor_scan_directory`

- ステップ
  - 指定ディレクトリ内の JPEG ファイルの集合を定期的に取得する。
  - 新規ファイルを検出したら、サイズが安定するまで待機し、`callback(file_path)` を呼び出す。
  - 一定時間「新規ファイルが来ていない」かつ「最低監視時間を満たした」場合などの条件で監視を終了する。
  - 処理したファイル数を返す。

### 正常系テスト

- 監視開始時にファイルが無い場合、何もコールバックされないまま終了する。
- 監視中に新しい JPEG ファイルが追加されると、`callback` がそのファイルパスで呼ばれる。
- しばらく新規ファイルが来なければ、終了条件を満たしてループを抜ける。

### 異常系テスト

- ディレクトリ読み取りで例外が発生した場合に、エラーメッセージを出しつつ監視を継続する。
- `callback` 内で例外が発生した場合に、そのファイルだけスキップし、ループ自体は継続する。

## `batch_scan_with_scanimage`

- ステップ
  - `mode.json` から指定モード名の設定を取得し、タイムスタンプ付きの tmp ディレクトリを作成する。
  - スキャナ設定とモード設定からバッチ用 `scanimage` コマンドを組み立て、サブプロセスとして実行する。
  - 出力ディレクトリ内の JPEG を列挙し、必要に応じて白紙除外などの後処理を行う。
  - `file_format` が PDF の場合は `convert_images_to_pdf` で PDF を生成し、Uploader 経由でクラウドにアップロードする。
  - 結果に応じて tmp ディレクトリの削除／パスの返却を行う。

### 正常系テスト

- 有効な `config_name` を指定したとき、組み立てられるバッチコマンド配列が期待どおりである（サブプロセスはモック）。
- `scanimage` が「N pages scanned」と出力し、N > 0 の場合に、mock した JPEG ファイルが存在し、関数が tmp ディレクトリパスを返す。
- `blank_filter=true` のモードで、白紙ページとみなしたファイル名が削除される（`is_blank_page` をテスト用にモック）。
- `file_format=PDF` のモードで PDF が生成され、Uploader の `upload_pdf` が呼ばれる。

### 異常系テスト

- `mode.json` に存在しない `config_name` を指定すると、エラーメッセージを出して `None` を返す。
- `scanimage` 実行がタイムアウトした場合に、例外をキャッチしつつ「生成されたファイルがあれば後続処理は進む」パスを通る。
- JPEG が 0 枚（`jpg_files` 空）の場合に、警告を出し、tmp ディレクトリを削除して `None` を返す。

---

## `single_scan_with_scanimage`

- ステップ
  - `mode.json` から指定モードの単発スキャン設定を取得する。
  - `ScannerManager` でスキャナをウォームアップする。
  - 設定に基づき単発スキャン用 `scanimage` コマンド配列を組み立てて実行する。
  - 成功時に出力ファイルの存在を確認し、必要に応じて Uploader でクラウドにアップロードする。
  - タイムアウト時はファイルの存在を確認し、あれば成功として扱う。

### 正常系テスト

- 有効な `config_name` と `output_path` を指定したときに、組み立てられるコマンド配列が期待どおりである。
- `scanimage` が成功した場合に True を返し、`upload_to_nextcloud=True` のとき Uploader の `upload_file` が呼ばれる。

### 異常系テスト

- `config_name` が `mode.json` に存在しない場合に False を返す。
- `ScannerManager.warm_up_scanner()` が False を返した場合に、スキャンを行わず False を返す。
- `scanimage` 実行タイムアウト時に、ファイルが存在すれば True、存在しなければ False を返す。

---

## `ScannerManager`

- ステップ
  - 初期化時に `scanner.json` からデバイス名・デフォルト設定を読み込む。
  - `check_scanner_available()` で `scanimage -L` の出力からデバイス存在を確認し、必要ならデバイス名を更新する。
  - `warm_up_scanner()` で簡単な自己診断コマンドを実行し、スキャナが応答するかを確認する。
  - シングルトンとして `get_instance()` から同じインスタンスを返す。

### 正常系テスト

- `get_instance()` を複数回呼んでも同じインスタンスが返る（シングルトン）。
- `check_scanner_available()` が、`scanimage -L` 出力に `device_name` が含まれているケースで True を返す。
- `warm_up_scanner()` が、自己診断コマンド成功時に True を返す。

### 異常系テスト

- `scanimage -L` に `device_name` が含まれていないが、出力に `fujitsu` + `ix500` が含まれる場合に、`device_name` が更新され True を返す。  
  ※ 現状は iX500 固定のハードコード仕様であり、将来的には「ベンダー名／モデル名を設定から読む」形に抽象化し、テスト観点もそれに合わせて汎用化する余地がある。
- `scanimage -L` 実行時に例外が投げられた場合にも、ログを出しつつ True を返し、固定 `device_name` を使い続ける。
- 自己診断コマンドが非 0 の戻り値を返した場合に、エラー出力をログに残しつつ `warm_up_scanner()` 自体は True を返す。

## `load_nextcloud_config`

- ステップ
  - `upload.json` を読み込んで dict に展開する。
  - `provider` フィールドを確認し、`"nextcloud"` 以外なら警告を出す。
  - `nextcloud` セクションを取り出し、辞書でなければ `ValueError` を送出する。

### 正常系テスト

- `provider="nextcloud"` かつ適切な `nextcloud` セクションを持つ `upload.json` から、期待どおりの dict が返る。

### 異常系テスト

- `provider` に別の値（例: `"dropbox"`）が入っていても、警告を出しつつ `nextcloud` セクションが返る。
- `nextcloud` セクションが存在しない／辞書でない場合に `ValueError` が送出される。

## `upload_file_to_nextcloud`

- ステップ
  - ローカルファイルの存在を確認する。
  - `remote_path` が無い場合、`upload_folder + ファイル名` でリモートパスを決定する。
  - `curl -X PUT` コマンドを組み立てて実行する。
  - stdout から HTTP ステータスコードを抽出し、2xx のとき成功とみなす。ステータス行が無い場合は戻り値や stderr から成否を推測する。

### 正常系テスト

- ローカルファイルが存在し、HTTP 2xx が返るケースで True を返す。
- `remote_path=None` のとき、`upload_folder + ファイル名` が使われる。

### 異常系テスト

- ローカルファイルが存在しないときに False を返す。
- HTTP 4xx/5xx のステータスコードが返るときに False を返す。
- ステータス行が無くても、戻り値 0 かつ stderr に `100.0%` を含む場合は True を返す。
- それ以外（戻り値非 0、または `100.0%` が無い）場合に False を返す。

---

## `upload_directory_to_nextcloud`

- ステップ
  - ディレクトリの存在を確認し、`delete_after_upload` を設定から決定する。
  - ディレクトリ名を使ってリモートサブディレクトリ `remote_dir` を作る。
  - `create_remote_directory` でリモートディレクトリを作成する。
  - ローカルディレクトリ内の全ファイルを `upload_file_to_nextcloud` でアップロードし、成功したものを記録する。
  - 全体成功かつ `delete_after_upload=True` のとき、ローカルファイルと空ディレクトリを削除する。

### 正常系テスト

- ディレクトリ内の全ファイルが `upload_file_to_nextcloud` を通じてアップロードされる。
- `delete_after_upload=True` の場合、アップロード成功後にローカルファイルと空ディレクトリが削除される。

### 異常系テスト

- ディレクトリが存在しないときに False を返す。
- ディレクトリ内にファイルが 0 件のときに False を返す。
- 1 ファイルでもアップロード失敗があるときに、`upload_success` が False になるが、成功したファイルはアップロードされている。

---

## `upload_pdf_to_nextcloud`

- ステップ
  - PDF ファイルの存在を確認する。
  - `delete_after_upload` を設定から決定する。
  - `upload_folder + ファイル名` をリモートパスとして `upload_file_to_nextcloud` を呼び出す。
  - アップロード成功かつ `delete_after_upload=True` の場合、ローカル PDF を削除する。

### 正常系テスト

- PDF ファイルが存在し、HTTP 2xx 相当が返るときに True を返す。
- `delete_after_upload=True` の場合、アップロード成功後にローカル PDF が削除される。

### 異常系テスト

- PDF ファイルが存在しないときに False を返す。
- アップロード失敗時に False を返し、ローカル PDF が削除されない。

## `get_uploader_from_config`

- ステップ
  - `upload.json` を読み込み、`provider` を取得する。
  - `provider` に応じて適切な Uploader インスタンスを生成する。
  - 未サポートの `provider` であれば警告を出しつつ `NextcloudUploader` にフォールバックする。

### 正常系テスト

- `provider="nextcloud"` のときに `NextcloudUploader` インスタンスを返す。

### 異常系テスト

- `provider` に未サポート文字列（例: `"dropbox"`）が設定されている場合に、警告を出しつつ `NextcloudUploader` を返す。

## `load_scan_modes`

- ステップ
  - `mode.json` を読み込み、`keybindings` セクションを取得する。
  - `keybindings` が dict かつ非空ならそれを返す。
  - ファイルが無い／`keybindings` が無い／例外発生時はデフォルト `{ "1": "diary", "2": "receipt", "3": "flyer" }` を返す。

### 正常系テスト

- `mode.json.keybindings` が存在し、期待どおりのマップが返る。
- `keybindings` が空 or 無いときに、デフォルトマップが返る。

### 異常系テスト

- `mode.json` が存在しない／読み込み時に例外が起こる場合に、エラーをログ or print しつつデフォルトマップが返る。

## `decode_key_name`（将来のヘルパー関数案）

- ステップ
  - `evdev.ecodes.KEY` マップから `key_code` に対応するキー名を取得する。
  - 未知のコードに対しては、例外を投げずにフォールバック表現を返す（仕様次第で `None` か特定文字列）。

### 正常系テスト

- 既知の `key_code` に対して `KEY_KP1` など正しいキー名が返る。

### 異常系テスト

- マップに存在しないコードが来た場合に、例外ではなく安全なフォールバック（`None` など）を返す。

---

## `extract_digit_from_key_name`（将来のヘルパー関数案）

- ステップ
  - 文字列が `"KEY_KP<digit>"` 形式であれば `<digit>` 部分を取り出す。
  - それ以外の形式であれば `None` を返す。

### 正常系テスト

- `"KEY_KP1"` → `"1"` を返す。
- `"KEY_KP9"` → `"9"` を返す。

### 異常系テスト

- `"KEY_ENTER"` や `"KEY_A"` を渡したときに `None` を返す。

---

## `resolve_mode_from_input`（将来のヘルパー関数案）

- ステップ
  - `scan_modes` dict（例: `{ "1": "diary", "2": "receipt" }`）から、`input_key` に対応する値を探す。
  - 存在すればモード名を返し、無ければ `None` を返す。

### 正常系テスト

- `input_key="1"` かつ `{"1": "diary"}` のとき `"diary"` を返す。

### 異常系テスト

- `input_key` が dict に存在しない（例: `"9"`）場合に `None` を返す。
